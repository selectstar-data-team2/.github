# 셀렉트스타 데이터 이행 2팀 GitHub 활용 가이드

## 레포지토리 구분
- **data-team2-foundation-code** 레포지토리는 데이터이행2팀이 공동으로 사용하는 중앙화 코드를 관리합니다.
- 그 외 프로젝트 진행을 위한 레포지토리가 필요한 경우, **프로젝트 당 하나의** 레포지토리를 만들어 활용합니다.
- 프로젝트 종료 시, 중앙화 코드에 반영이 필요한 내용이 있다면 팀에 공유하여 중앙화 코드에 추가합니다.

## 레포지토리 생성
---
- **Repository name**: 채번 사용 (예: `25-PD-005-b_kt`)
- **Description**: 프로젝트 및 레포지토리 설명 작성
- **Visibility**: `private`
- **README**: `on`
- **Add .gitignore**: `No gitignore`
- **Add license**: `No license`
- `Create repository` 클릭

## README.md 작성 템플릿
* 템플릿 파일: [README_TEMPLATE.md](README_TEMPLATE.md)

## 브랜치 네이밍
---
**형식**: `[종류]/[생성자 이름]`

| 종류 | 의미 | 예시 |
|------|------|------|
| `feat/` | 새 기능 추가 | `feat/hong` |
| `fix/` | 오류 수정 | `fix/kim` |
| `chore/` | 파일 정리 | `chore/park` |
| `docs/` | 문서 수정 | `docs/choi` |

## 커밋 메시지
---
**형식**: `[종류]/[업무 내용에 대해 한글로 상세히]`

| 종류 | 의미 | 예시 |
|------|------|------|
| `feat` | 새 기능 추가 | `feat/네이버 뉴스 크롤러 추가` |
| `fix` | 오류 수정 | `fix/엑셀 한글 깨짐 수정` |
| `chore` | 파일 정리 | `chore/임시 파일 삭제` |
| `docs` | 문서 수정 | `docs/사용 가이드 추가` |


### Git 커밋 매너

**나의 작업을 알 수 있게 작성해주기**

- 나중에 히스토리를 볼 때 `feat: 수정`이라고 되어 있으면 무엇을 고쳤는지 아무도 알 수 없습니다.
- `feat: 메인 기능 이미지 교체` (O)

**한 번에 한 일만 커밋하기**
- 오타도 고치고, 새 기능도 만들고, 파일도 지웠다면? 각각 커밋하는 게 좋지만, 어렵다면 가장 **비중이 큰 작업**을 기준으로 적어주세요.

**작업 단위가 끝날 때마다 '저장' 버튼이라 생각하기**
- 퇴근할 때 몰아서 한 번에 커밋하면 나중에 문제가 생겼을 때 되돌리기가 매우 힘듭니다.

## 이슈 생성
---

**이슈를 생성하는 경우:**
- 당장 고칠 수 없는 에러
- 나중에 수정, 추가 개발할 기능
- 다른 담당자에게 공유가 필요한 특이사항 존재

**사용 가능한 템플릿:**
- **버그 리포트**: 버그를 보고할 때 사용
- **기능 제안**: 새로운 기능을 제안할 때 사용
